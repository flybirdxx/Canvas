# 项目架构重构建议

## 目标

本建议用于指导 `Canvas` 项目的维护型架构拆解。目标不是重写项目，也不是新增或删除功能，而是在保持现有行为稳定的前提下，逐步降低模块耦合，让后续节点类型、AI 能力、画布交互和执行流程可以更低风险地迭代。

当前策略是：先按新版方案建立能力边界，等耦合明显下降并完成验证后，再吸收旧版方案中 `core / application / infrastructure / presentation` 的分层思想做目录重组。

## 当前主要耦合点

1. 节点能力定义分散在类型、端口、渲染、快捷键、连接、flow resolver 和执行引擎中。
2. `InfiniteCanvas.tsx` 同时承载 Stage 装配和多个 pointer 事件优先级判断。
3. `CanvasElements.tsx` 同时负责节点遍历、渲染分发、拖拽、选择、resize、端口和运行状态装配。
4. 端口布局和命中逻辑缺少单一来源。
5. 执行引擎直接知道过多节点业务语义、全局 store 和 UI toast。
6. gateway 的 provider 注册、价格、配置读取和能力分发曾集中在单文件中。
7. `flowResolver.ts` 曾逐步变成隐形节点 registry。
8. OmniScript 的 prompt、parser、analyzer 和 UI 上游媒体解析边界不清晰。

## 新版能力边界执行状态

| 阶段 | 目标 | 当前状态 |
| --- | --- | --- |
| 1 | 建立节点能力注册表 `nodeRegistry` | 已落地 |
| 2 | 统一节点创建入口 `createCanvasElement` | 已落地 |
| 3 | 抽出画布 `useCanvasPointerController` | 已落地 |
| 4 | 统一端口布局和命中 `getPortLayout` | 已落地 |
| 5 | 下沉 resize 几何规则 | 已落地 |
| 6 | 抽出节点执行策略 `nodeExecutionPolicy` | 已落地 |
| 7 | 拆分 gateway 与 RunningHub provider | 已落地，需修复 review 发现的问题 |
| 8 | 拆分 OmniScript 编排和媒体解析 | 已落地，需修复 review 发现的问题 |

结论：新版能力边界流程已经进入收尾修复阶段。暂不继续扩大到 old 方案的大规模目录分层重组，避免在当前修复完成前制造额外 import churn。

## Diff Review 留存问题

以下问题来自全覆盖 diff review，进入修复流程前必须保留并逐项验证：

1. `src/services/gateway/providers/runninghub/video.ts` 中视频轮询成功结果被再次传给 `extractUrls`，导致 `{ ok: true, urls }` 被误判为空结果。
2. RunningHub 视频提交阶段的异常可能直接抛出到 `runVideoGeneration`，破坏“返回结构化失败”的约定。
3. `src/services/omniscript/prompt.ts` 出现乱码和插值损坏，用户 notes 没有进入模型 prompt。
4. `src/registry/createCanvasElement.ts` 和 `src/registry/nodeRegistry.ts` 的默认中文文案乱码，影响新建节点和后续 registry UI 来源。
5. gateway / RunningHub 多处结构化错误消息乱码或插值损坏，导致错误信息不可读或缺少模型、provider、服务端 message。
6. `src/utils/flowResolver.ts` 的上游文本贡献 label 插值损坏，来源标签缺少首行摘要。
7. 需要补充针对 RunningHub 视频成功轮询、视频提交失败结构化返回、OmniScript notes、节点默认中文文案、gateway 错误消息插值的回归测试。

## 后续执行原则

1. 当前先修复 diff review 留存问题，并用测试固定边界契约。
2. 修复完成后运行 `npm run lint:tsc` 和 `npm run test`。
3. 画布交互相关改动需要补充浏览器行为验证。
4. `.superpowers/` 是本地工具目录，不进入提交范围。
5. 完成本轮修复后，再评估是否进入 old 方案吸收阶段。
